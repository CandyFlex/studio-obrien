## (d) Recommended adjustments to INTEL-SYSTEM Section 2 weekly routine

Based on what this first live run actually showed:

1. **Reorder the sweep to lead with the feeds that work headless.** Section 2's table lists SoS New/Dissolved as feed #1, but it is the one feed that is fully bot-blocked. Put the **env-health portal and Chamber events calendar first** (both fetchable, both fresh), and treat SoS as a manual browser step done alongside. Do not let the blocked #1 stall the sweep.

2. **Downgrade SoS New/Dissolved to an explicit "Jarred-manual, browser-only" step.** Record in Section 2 that automated fetch returns 403 on both the New/Dissolved and general entity-search views. The weekly routine should budget ~5 min of Jarred's browser time for it and confirm the county/agent-address filter on that first visit (still `unverified`). Consider whether the paid Data Subscriptions program is worth it *only if* the browser view proves too coarse — do not pay before testing the free view manually once.

3. **Upgrade the Chamber feed from "manual only" to "calendar-automatable, FB-manual."** Section 1 currently says the Chamber is "a manual weekly look" with "no verified feed." Correction: the **events calendar at business.clevelandchamber.org/events is public and fetchable** and lists ribbon cuttings (named + placeholder). Only new-member announcements and unnamed placeholder cuttings still require the login-gated FB page. Split the note accordingly.

4. **Add a "first-inspection ≠ flagged" caveat to the env-health feed.** The portal is date-sortable but never labels a row as a *first* inspection, so "new restaurant" is inferred. The routine should add one per-row confirmation step (Google/Maps age + inspection history) before promoting an env-health row to a fresh lead — otherwise routine re-inspections of McDonald's/Subway/Starbucks read as noise. Sorting by date is necessary but not sufficient.

5. **Add a web-status gate before assigning the strike read.** This pass showed the wait-vs-strike rule alone is not enough: three of five food hits (Chubby's, Rhodies, Thelma Lou's) *already had sites*, which changes the play from "no-site build" to "site-quality upgrade or review-ask." Recommend the routine record web status (none / social-only / DIY-thin / good) as a required field on every fresh-lead row, and branch: **social-only or none = strike (build)**, **DIY-thin = strike (upgrade angle)**, **good site = drop or review-ask only**. The scorecard's gap factor implies this; Section 2 should state it explicitly for L4 rows.

6. **Kings Mountain is where the fresh signal is concentrated right now.** Every live env-health/news hit this pass was KM (casino-corridor). Worth noting in the routine that the Two-Kings growth is producing real new-storefront volume in a Tier 2 town — the saturation caps (Part 5e, max 5 openers/month per 8k–15k town) will bind here fast, so prioritize the best 1–2 KM leads rather than working all of them.
